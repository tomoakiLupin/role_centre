# Role Center Bot 使用文档

## 1. 简介

Role Center Bot 是一个功能强大的 Discord 机器人，旨在自动化身份组管理、申请处理和服务器统计等任务。它通过斜杠命令 (`/`) 与用户交互，并集成了 gRPC 服务以实现更复杂的功能。

## 2. 安装与启动

### 2.1. 环境要求

*   Node.js (建议 v16 或更高版本)
*   pnpm

### 2.2. 安装步骤

1.  **克隆项目**
    ```bash
    git clone <your-repo-url>
    cd role_center
    ```

2.  **安装依赖**
    ```bash
    pnpm install
    ```

3.  **配置环境变量**
    *   复制 `.env.example` 文件并重命名为 `.env`。
    *   打开 `.env` 文件，填入你的 Discord Bot Token。
    
    ```env
    # Discord Bot Token
    DISCORD_TOKEN=你的Bot令牌

    # 是否在启动时运行扫描 (true/false)
    RUN_SCAN_ON_STARTUP=false
    ```

### 2.3. 启动机器人

运行以下命令来编译 gRPC 文件并启动机器人：

```bash
npm start
```

如果一切顺利，你的机器人现在应该已经在线了。

## 3. 配置说明

机器人的核心配置位于 `/config` 目录下的多个 `.json` 文件中。系统会自动加载这些文件，并且支持通过环境变量进行覆盖。

### 3.1. 主要配置文件

*   `bot_config.json`: 机器人的基本配置，例如客户端 ID。
*   `log_config.json`: 日志记录相关的配置。
*   `blacklist.json`: 用户黑名单。
*   `channle_san_config.json`: 频道扫描相关配置。
*   `atuo_applyrole_config.json`:自动身份组申请相关配置。
*   `post_reaction_autoapply.json`: 通过帖子反应自动申请身份组的配置。

### 3.2. 使用环境变量覆盖配置

你可以通过设置特定格式的环境变量来覆盖 JSON 文件中的配置项。格式为 `CONFIG_文件名_配置键`。

**示例**:
要覆盖 `bot_config.json` 中的 `clientId`，可以设置环境变量：
`CONFIG_BOT_CLIENTID="your_client_id"`

## 4. 命令参考

以下是 Role Center Bot 支持的所有斜杠命令 (`/`) 的详细说明。

### 4.1. 管理与面板创建

#### `/创建自主获取面板`
*   **功能**: 创建一个带有按钮的面板，用户可以通过点击按钮来获取指定的身份组。
*   **参数**:
    *   `role_id` (必填): 要分配的身份组的 ID。
    *   `title` (必填): 面板的标题。
    *   `content` (必填): 面板的内容描述。
    *   `image_url` (可选): 在面板中显示的图片 URL。
    *   `channel` (可选): 发送面板的频道，默认为当前频道。

#### `/创建身份组申请面板`
*   **功能**: 基于 `atuo_applyrole_config.json` 中的配置，创建一个身份组申请面板。
*   **参数**:
    *   `config_id` (必填): 配置文件中的配置 ID，支持自动补全。
    *   `title` (可选): 自定义面板标题。
    *   `description` (可选): 自定义面板描述。
    *   `channel` (可选): 发送面板的频道，默认为当前频道。

#### `/创建退出面板`
*   **功能**: 创建一个面板，用户可以通过按钮退出一个或多个身份组。
*   **参数**:
    *   `role_ids` (必填): 一个或多个身份组的 ID，使用逗号分隔。
    *   `enable_logging` (可选): 是否生成退出日志，默认为 `false`。
    *   `auto_delete_minutes` (可选): 面板自动删除的时间（分钟），`0` 表示不删除。

#### `/创建身份组管理面板`
*   **功能**: (管理员权限) 创建一个面板，让用户可以管理自己拥有的、通过机器人获取的身份组。
*   **参数**:
    *   `title` (可选): 面板的标题。
    *   `content` (可选): 面板的内容描述。
    *   `image_url` (可选): 在面板中显示的图片 URL。
    *   `channel` (可选): 发送面板的频道，默认为当前频道。

#### `/create_post_apply_panel`
*   **功能**: (管理员权限) 创建一个基于帖子反应的身份组申请面板。当一个帖子达到指定的反应数时，发帖人可以获得身份组。
*   **参数**:
    *   `role_id` (必填): 需要申请的身份组 ID。
    *   `reactions` (必填): 帖子需要达到的最高反应数。
    *   `channel` (可选): 帖子所在的论坛频道，默认为所有频道。
    *   `grpc_config_id` (可选): gRPC 配置 ID，用于启用远程数据库查询。
    *   `grpc_button_name` (可选): gRPC 查询按钮的名称。

### 4.2. 用户功能

#### `/查询我的消息数据库`
*   **功能**: 查询用户在消息数据库中的统计数据。
*   **参数**:
    *   `user` (可选): (管理员权限) 查询指定用户的统计数据。

### 4.3. 批量操作

#### `/批量分发`
*   **功能**: 批量将一个或两个身份组分发给指定的用户列表或参与了某个消息的用户。
*   **参数**:
    *   `role_id_1` (必填): 第一个身份组 ID。
    *   `role_id_2` (可选): 第二个身份组 ID。
    *   `user_ids` (可选): 用户 ID 列表，用逗号分隔。
    *   `message_link` (可选): 消息链接，机器人会自动解析参与该消息的用户。
    *   `operation_id` (可选): 用于追加到现有操作的 ID。
    *   `timeout` (可选): 身份组的有效期（分钟），默认 90 分钟。
    *   `skip_auto_expire` (可选): 是否跳过自动过期，默认为 `false`。


## 5. 核心功能与交互

本节将详细介绍机器人的主要功能以及用户如何与之交互。

### 5.1. 自主获取身份组

通过 `/创建自主获取面板` 命令，管理员可以轻松创建一个面板，让服务器成员可以自助获取或退出某个特定的身份组。

*   **流程**:
    1.  管理员使用命令创建一个包含“加入”和“退出”按钮的面板。
    2.  用户点击按钮，机器人会立即为其添加或移除相应的身份组。
    3.  操作结果会以临时消息的形式反馈给用户。

### 5.2. 身份组退出面板

当需要让用户一次性退出多个身份组时，可以使用 `/创建退出面板` 命令。

*   **流程**:
    1.  管理员定义一组可退出的身份组，并创建一个退出面板。
    2.  用户点击“退出身份组”按钮。
    3.  机器人会检查用户拥有哪些可退出的身份组，并弹出一个确认窗口。
    4.  用户确认后，机器人将移除所有相关身份组。
    5.  管理员可以选择是否记录退出日志。

### 5.3. 身份组申请系统

这是一个高度可配置的复杂功能，允许用户根据一系列预设条件来申请身份组。

*   **申请流程**:
    1.  管理员通过 `/创建身份组申请面板` 命令，并关联一个在 `atuo_applyrole_config.json` 中定义的配置，来创建一个申请按钮。
    2.  用户点击申请按钮。
    3.  机器人开始验证：
        *   **黑名单/拒绝状态**: 检查用户是否被临时或永久禁止申请。
        *   **前置条件**: 检查用户是否拥有必需的前置身份组。
        *   **资格审查**: 通过 `db/eligibility_checker.js` 查询数据库，验证用户是否满足配置中定义的统计数据要求（例如，消息数量、特定频道发言次数等）。
    4.  **处理结果**:
        *   **自动批准**: 如果配置为自动模式且用户满足所有条件，机器人会自动授予身份组。
        *   **人工审核**: 如果配置为人工审核模式，即使用户满足条件，申请也会被提交到一个内部投票系统，由管理员进行审核。
        *   **拒绝**: 如果用户不满足任何条件，申请将被拒绝，并且用户会在一段时间内被禁止再次申请。

### 5.4. 基于帖子反应的申请

这是一个独特的功能，允许用户通过在论坛帖子中获得足够的反应来赢得身份组。

*   **流程**:
    1.  管理员使用 `/create_post_apply_panel` 命令设置一个目标身份组和所需的反应数量。
    2.  当一个论坛帖子的反应数达到阈值时，发帖人会收到一个通知，并可以通过按钮领取身份组。
    3.  此功能还支持通过 gRPC 查询外部数据库来验证资格。

### 5.5. 深入了解：人工审核与投票系统

当一个身份组申请被配置为需要人工审核时，机器人会启动一个复杂的投票流程。

*   **投票阶段**:
    1.  **发起**: 申请在指定的审核频道被创建为一个投票帖子，并 @ 相关的审核人员。
    2.  **用户投票**: 普通审核人员（拥有特定身份组）可以投“同意”或“拒绝”票。
    3.  **管理员确认期**: 一旦“同意”票达到预设门槛，投票会进入一个为期 24 小时的“管理员确认期”。在此期间，申请将等待最终批准。
*   **决策逻辑**:
    *   **直接拒绝**: 如果“拒绝”票（尤其是来自管理员的票）达到门槛，投票会立即失败。
    *   **自动批准**: 如果在 24 小时确认期内没有管理员投“拒绝”票，申请将自动通过。
    *   **手动批准**: 管理员可以在任何阶段投“同意”票来直接批准申请。
*   **结果**:
    *   **通过**: 用户获得身份组，并收到私信通知。
    *   **拒绝**: 用户会收到私信通知，并根据情况被临时或永久禁止再次申请该身份组。

### 5.6. 自动化与后台任务

机器人通过一系列定时任务来处理后台逻辑，确保系统平稳运行。

*   **身份组自动过期 (`/批量分发` 功能)**:
    *   当使用 `/批量分发` 命令并设置了有效期时，身份组信息会被记录在 `data/role_assignments.json` 中。
    *   一个每 3 小时运行一次的定时任务 (`role_fade_task.js`) 会检查此文件。
    *   对于已过期的身份组，机器人会自动将其移除，并替换为一个在 `bot_config.json` 中定义的“过期身份组”。

*   **投票自动批准**:
    *   一个每小时运行一次的定时任务 (`vote_auto_approve_task.js`) 会检查所有处于“管理员确认期”的投票。
    *   如果 24 小时已过，任务会自动批准这些申请。

*   **gRPC 集成**:
    *   机器人具备与外部微服务通过 gRPC 网关进行通信的能力。
    *   它既可以作为客户端向其他服务（如 `RecommendationService`）发起请求，也可以作为服务端响应来自网关的调用（如 `RoleService`）。
    *   这使得机器人可以轻松地与更广泛的系统生态集成，实现数据查询和功能扩展。
## 6. 高级配置详解

本节将深入介绍 `config/` 目录下的各个 JSON 配置文件，帮助您充分利用机器人的所有功能。

### 6.1. `bot_config.json` - 核心机器人配置

这个文件定义了机器人的核心行为和权限。

*   `safety_setting`:
    *   `developers_userid`: 机器人开发者的用户 ID 列表，拥有最高权限。
    *   `admin_roleid`: 管理员身份组的 ID 列表。
    *   `user_roleid`: 普通用户身份组的 ID 列表（用于投票等）。
    *   `command_push_guildids`: 机器人会向这些服务器注册斜杠命令。
*   `other`:
    *   `timeout_role_mapping`: 定义了当临时身份组过期时，应该替换为哪个“过期身份组”。每个服务器可以有不同的设置。
    *   `roleid_mapping_setup`: 用于特定功能的身份组映射。

### 6.2. `atuo_applyrole_config.json` - 身份组自动申请

这是机器人最复杂的配置文件，定义了所有身份组申请的规则。

*   **顶级键**: 每个键（如 `"1"`, `"2"`）代表一个独立的申请配置。
*   `name`: 配置的名称，会显示在命令的自动补全中。
*   `guild_id`: 此配置所属的服务器 ID。
*   `data`:
    *   `database_name`: 申请资格检查所依赖的数据库文件名（位于 `data/message_sacn/`）。
    *   `database_kv`: 要从数据库中检查的键名（如 `message_count`）。
    *   `threshold`: 用户必须达到的数值门槛。
    *   `musthold_role_id`: 申请前必须拥有的前置身份组 ID。
    *   `role_id`: 成功后授予的身份组 ID。
*   `manual_revive`: 如果为 `true`，则启用人工审核流程；否则为自动批准。
*   `revive_config`: 人工审核的详细配置。
    *   `review_channel_id`: 发起投票的频道 ID。
    *   `allow_vote_role`:
        *   `admin`: 管理员审核身份组 ID。
        *   `user`: 普通审核身份组 ID。
        *   `ratio_allow`: 通过所需的同意票数（分管理员和用户）。
        *   `ratio_reject`: 触发拒绝所需的反对票数。

### 6.3. `log_config.json` - 日志配置

定义了机器人日志的发送位置和样式。

*   `log_channel_id`: 所有日志消息都会被发送到这个频道。
*   `log_levels`: 定义了不同日志等级（如 `info`, `error`）的颜色和名称。

### 6.4. `channle_san_config.json` - 频道扫描配置

用于配置机器人的消息扫描功能，通常用于统计用户在特定频道的发言数量。

*   `channels_scan_config`:
    *   `db_path`: 扫描结果存放的 SQLite 数据库路径。
    *   `data`:
        *   `guilds_id`: 要扫描的服务器 ID。
        *   `channel_id`: 要扫描的频道 ID。
        *   `frist_message_id`: 从哪条消息开始扫描。

### 6.5. `blacklist.json` - 黑名单

用于禁止特定用户使用某些功能。

*   `btn`: 按钮交互的黑名单。键是按钮的 `customId` 前缀，值是用户 ID 列表。
*   `cmd`: 命令的黑名单。
